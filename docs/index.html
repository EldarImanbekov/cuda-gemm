// ABOUT ME

What Drives Me

There's something that genuinely excites me about working at the boundary between software and hardware. Most of the world uses AI as a black box — you type something in and something comes out. I want to understand the machinery that makes that possible, and more importantly, how to make that machinery run better.

Right now I'm focused on GPU programming and high-performance computing. I've been learning how memory works at the hardware level, why access patterns matter, and how to write code that actually takes advantage of what modern GPUs can do. It's a field where the gap between what hardware is capable of and what software actually uses is enormous — and closing that gap is the work I want to do.

Outside of the technical side, I think a lot about how to communicate hard ideas simply. The best engineers I've learned from can explain a complex system to anyone. That's a skill I work on deliberately.

CUDA · HPC · Performance

CUDA GEMM Optimization

I wanted to understand what actually happens when a GPU does math — not by reading about it, but by building it myself. So I wrote three versions of matrix multiplication from scratch: a naive baseline that reads from slow global memory, a tiled version that loads data into fast on-chip shared memory so threads can reuse it, and a coalesced version that fixes memory access patterns at the cache line level. Each one taught me something the previous one couldn't. Real benchmark numbers on a Tesla T4 GPU, with correctness verified against CPU reference implementation.
